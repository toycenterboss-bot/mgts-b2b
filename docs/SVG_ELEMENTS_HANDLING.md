# Обработка SVG элементов в нормализованном HTML

**Дата:** 2026-01-09  
**Статус:** ✅ SVG элементы обрабатываются корректно

## Анализ SVG элементов

### Что такое SVG элементы
SVG (Scalable Vector Graphics) элементы используются для отображения векторных иконок и графики на страницах. В нормализованном HTML встречаются:
- `<svg>` - корневой элемент SVG
- `<path>` - пути для отрисовки фигур
- `<g>` - группы элементов
- `<clipPath>` - пути для обрезки
- `<defs>` - определения (градиенты, паттерны и т.д.)

### Примеры SVG элементов в нормализованном HTML
```html
<svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" fill="none">
  <g fill="#000" clip-path="url(#Profile_svg__a)">
    <path d="M12 6.499a3.501 3.501 0 1 0 0 7.002 3.501 3.501 0 0 0 0-7.002"></path>
  </g>
  <defs>
    <clipPath id="Profile_svg__a">
      <path fill="#fff" d="M0 0h24v24H0z"></path>
    </clipPath>
  </defs>
</svg>
```

## Текущая обработка SVG элементов

### ✅ Что уже обрабатывается
1. **Классы на SVG элементах:**
   - Классы на `<svg>` элементах (например, `link-img`, `file-item__type-img`) преобразуются в целевые классы компонентов
   - Классы на `<g>` элементах обрабатываются через маппинг

2. **Внешние классы:**
   - Классы, начинающиеся с `Logo_svg__`, `__cls-` сохраняются как есть (используются внешними библиотеками)
   - Классы Яндекс карт (`ymaps3x0--`) сохраняются как есть

### ⚠️ Что НЕ обрабатывается (и это правильно)
1. **Атрибуты SVG элементов:**
   - `d` (путь), `fill`, `stroke`, `clip-path`, `viewBox` и другие SVG-специфичные атрибуты остаются как есть
   - Это правильно, так как они являются частью векторной графики

2. **Внутренняя структура SVG:**
   - `<path>`, `<g>`, `<clipPath>`, `<defs>` элементы остаются без изменений
   - Это правильно, так как они определяют форму иконки

## Нужно ли что-то делать с SVG элементами?

### ✅ НЕТ, ничего делать не нужно

**Причины:**
1. **SVG элементы - это контент, а не стили:**
   - SVG определяет форму иконки, а не стили
   - Атрибуты `d`, `fill`, `stroke` являются частью векторной графики
   - Они не требуют нормализации, так как не являются CSS классами

2. **SVG элементы уже правильно обработаны:**
   - Классы на SVG элементах преобразуются через маппинг
   - Внешние классы (Logo_svg__, __cls-) сохраняются как есть
   - Внутренняя структура SVG остается без изменений

3. **SVG элементы будут работать на новом сайте:**
   - SVG - это стандартный формат, поддерживаемый всеми браузерами
   - При миграции SVG элементы будут работать без изменений
   - Единственное, что может потребоваться - это обновление путей к SVG файлам, если они хранятся отдельно

## Рекомендации

### ✅ Оставить как есть
- SVG элементы не требуют дополнительной обработки
- Они уже правильно обработаны в скрипте нормализации
- При миграции они будут работать без изменений

### ⚠️ Что проверить при миграции
1. **Пути к SVG файлам:**
   - Если SVG хранятся в отдельных файлах, нужно обновить пути
   - Если SVG встроены в HTML (inline SVG), они будут работать как есть

2. **CSS для SVG:**
   - Убедиться, что CSS классы на SVG элементах правильно стилизуются
   - Проверить, что классы типа `link-img`, `file-item__type-img` имеют соответствующие стили

3. **Размеры SVG:**
   - Проверить, что атрибуты `width` и `height` корректны
   - Убедиться, что SVG масштабируются правильно на разных экранах
